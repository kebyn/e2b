## 阅读导航

### 如果你在做最小部署

PostgreSQL 和 Redis 不能省；API 可用 `SERVICE_DISCOVERY_PROVIDER=local` 绕过 Nomad，Orchestrator 可用 `USE_LOCAL_NAMESPACE_STORAGE=true` 绕过 Consul，LaunchDarkly 和 ClickHouse 可以不配置。

### 如果你在做多节点部署

选择 Nomad 或 Kubernetes 作为 discovery/调度基础；Redis 和 PostgreSQL 必须具备高可用入口。是否保留 Consul 取决于 Orchestrator 的 namespace storage 选择。

### 如果你要部署 Dashboard API

除 PostgreSQL 和 Redis 外，还必须配置 `ADMIN_TOKEN`、`ORY_SDK_URL`、`ORY_PROJECT_API_TOKEN` 和可验证的 Ory issuer。

---

## 总览

| 组件 | 单节点 | 多节点 / HA | 当前无代码替代 |
|------|--------|-------------|----------------|
| PostgreSQL | 必须 | 必须 | 无 |
| Redis | API、Client Proxy 必须 | 必须 | 支持单节点地址或单个 Cluster 引导端点 |
| Auth Provider | API Key 场景可不配 | 用户级 JWT/OIDC 时需要 | `AUTH_PROVIDER_CONFIG` 可接外部 issuer |
| Ory | 不部署 Dashboard API 时可省 | Dashboard API 必须 | 无通用 profile provider 替代开关 |
| Nomad | 可省 | 默认部署路径 | Kubernetes discovery；`local` 仅单节点 |
| Consul | 可省 | 取决于 namespace storage | `USE_LOCAL_NAMESPACE_STORAGE=true` 使用本机状态 |
| LaunchDarkly | 可省 | 可省 | 未配置时使用源码 fallback |
| ClickHouse | 可省 | 可选 | 未配置时相关写入使用空实现 |

## 1. Auth Provider 与 Ory

### 当前能力

- API 和 Dashboard API 从 `AUTH_PROVIDER_CONFIG` 读取 JWT/OIDC issuer 配置。
- 旧的 `SUPABASE_JWT_SECRETS` 不属于 `2026.28` 当前配置契约。
- 不配置 `AUTH_PROVIDER_CONFIG` 时，外部 JWT/OIDC 登录不可用；API Key、尚未被 Feature Flag 禁用的旧 access token 和 admin token 路径仍由各自 authenticator 处理。
- Dashboard API 固定依赖 Ory profile/identity 能力，不能仅用 `AUTH_PROVIDER_CONFIG` 替代。

### Dashboard API 条件

Dashboard API 启动时要求：

| 配置 | 要求 |
|------|------|
| `POSTGRES_CONNECTION_STRING` | 必须且非空 |
| `ADMIN_TOKEN` | 必须且非空 |
| `REDIS_URL` / `REDIS_CLUSTER_URL` | 至少一个 |
| `ORY_SDK_URL` | 必须且非空 |
| `ORY_PROJECT_API_TOKEN` | 必须且非空 |
| `ORY_ISSUER_URL` | 通常必须；单一 JWT issuer 时可推导 |

如果 `AUTH_PROVIDER_CONFIG.jwt` 非空，最终的 `ORY_ISSUER_URL` 必须与其中一个 `issuer.url` 完全匹配。

### 取舍

- 仅内部 API Key 调用、不部署 Dashboard：可不配置外部 Auth Provider 和 Ory。
- 面向用户的 Dashboard：保留外部 OIDC/JWT provider 和 Ory。
- Supabase 可作为外部身份系统之一，但当前部署只能通过通用 provider 配置接入，不能继续使用旧专用变量。

## 2. 服务发现与调度

`SERVICE_DISCOVERY_PROVIDER` 默认值是 `nomad`，合法值只有 `nomad`、`kubernetes` 和 `local`。

### Nomad

- `NOMAD_ADDRESS` 默认 `http://localhost:4646`。
- `NOMAD_ORCHESTRATOR_SERVICE_NAMES` 默认 `orchestrator`。
- `NOMAD_ORCHESTRATOR_LEGACY_DISCOVERY_ENABLED` 默认 `true`，用于合并旧 node-pool 回退。
- 省略 `NOMAD_ADDRESS` 不会自动切换到 local provider。

Nomad 是上游 `self-host.md` 和官方 jobspec 的默认部署路径，同时承担 workload 调度。API discovery 通过 Nomad native service registration 获取 Orchestrator，再用 gRPC 健康探测筛选节点。

### Kubernetes

使用内置 Kubernetes discovery 时：

`SERVICE_DISCOVERY_PROVIDER=kubernetes`

默认 selector 为：

- Orchestrator：`app.kubernetes.io/name=orchestrator`
- Template Manager：`app.kubernetes.io/name=template-manager`

API ServiceAccount 必须能在目标 namespace 对 Pod 执行 `get`、`list`、`watch`。当前 discovery 从 Pod `HostIP` 组合固定的 `5008` 地址，因此 Orchestrator 和 Template Manager 的部署必须满足 `hostNetwork=true` 及固定端口假设。

### Local

`SERVICE_DISCOVERY_PROVIDER=local`
`LOCAL_ORCHESTRATOR_ADDRESS=127.0.0.1:5008`

local provider 只返回一个静态 Orchestrator，适合本地开发和单节点验证，不是 HA discovery。

### 不存在的 provider

`2026.28` 没有 static-list、Redis 或文件型 discovery provider。新增这类 provider 需要修改 API 配置校验、初始化分支和实现代码，不能通过写一个未定义的环境变量启用。

## 3. Consul 与 Namespace Storage

Consul 只用于 Orchestrator sandbox network slot reservation。当前实现位于：

- `infra/packages/orchestrator/pkg/sandbox/network/storage_kv.go`
- `infra/packages/orchestrator/pkg/sandbox/network/storage_local.go`
- `infra/packages/orchestrator/pkg/factories/run.go`

### 当前选择逻辑

- `ENVIRONMENT=dev`、`ENVIRONMENT=local` 或 `USE_LOCAL_NAMESPACE_STORAGE=true`：使用本机 namespace 状态。
- 生产默认且未开启 local storage：使用 Consul KV，需要 `CONSUL_TOKEN`。

Consul key 包含 `nodeID`。它提供持久的 reservation 和并发控制，但不应描述成“所有节点共享同一个 IP 池”。

### 多节点取舍

- 每台 Orchestrator 独占宿主机网络命名空间时，可显式启用 local namespace storage；所选 CIDR 必须避免与该宿主机、Pod 网络和现有路由冲突。
- 使用上游默认网络配置时，保留 Consul KV。
- local storage 不跨节点共享状态；节点替换和遗留 namespace 回收必须纳入运维流程。

### 不存在的 Redis slot backend

`2026.28` 没有 `IP_SLOT_STORAGE`，也没有 Redis 实现的 network slot storage。把 `IP_SLOT_STORAGE=redis` 写进环境不会产生任何效果。实现 Redis backend 属于运行时代码改造，不是本次文档同步或“无代码部署”能力。

## 4. Feature Flags

当前只内置 LaunchDarkly client 和本地 offline store：

- 配置 `LAUNCH_DARKLY_API_KEY`：连接 LaunchDarkly。
- 不配置：返回 `flags.go` 注册的 fallback。
- 没有 `FEATURE_FLAGS_PROVIDER`，也没有 YAML 或 Unleash provider。

因此 LaunchDarkly 不是启动必需组件。需要动态灰度时保留它；不需要动态覆盖时使用 offline fallback。当前 80 个 Flag 的名称、类型和默认行为见 [`FeatureFlags私有化部署方案.md`](../feature-flags/FeatureFlags私有化部署方案.md)。

## 5. Redis

### 必需性

| 服务 | Redis 行为 |
|------|------------|
| API | 必须；创建 client 失败会终止启动 |
| Client Proxy | 必须；Sandbox Catalog 固定使用 Redis |
| Dashboard API | `REDIS_URL` / `REDIS_CLUSTER_URL` 至少一个 |
| Orchestrator | 可选；未配置时禁用 Redis event delivery 和 peer registry |

API 和 Client Proxy 不存在内存 catalog 降级，即使是单节点开发也必须提供 Redis。

### 地址契约

- `REDIS_URL`：单个 `host:port`。
- `REDIS_CLUSTER_URL`：单个 Redis Cluster 引导端点，不是逗号分隔列表。
- 两者都配置时，Cluster 配置优先。
- 应用不支持 Sentinel/Failover client；Sentinel 主从应在前面提供 HAProxy、VIP 或云负载均衡入口。

连接池默认值：

| 服务 | `REDIS_POOL_SIZE` | `REDIS_MIN_IDLE_CONNS` |
|------|-------------------|------------------------|
| API | 160 | 不读取 |
| Client Proxy | 40 | 不读取 |
| Orchestrator | 5 | 2 |

## 6. PostgreSQL 与 Migrations

API 和 Dashboard API 的 `POSTGRES_CONNECTION_STRING` 都是必需配置。正式部署应先运行固定子模块中的 migrator：

~~~bash
make -C infra/packages/db migrate
~~~

PostgreSQL migration tracking 表是 `_migrations`。官方 Nomad API job 使用 `infra/packages/db/Dockerfile` 构建的 `db-migrator` prestart task，入口为 `./migrator`。

---

## 7. ClickHouse

`CLICKHOUSE_CONNECTION_STRING` 为空时可不启用 ClickHouse；`CLICKHOUSE_CONNECTION_STRINGS` 使用分号分隔附加端点，并受 `clickhouse-read-endpoint` 和 `clickhouse-write-fanout` Feature Flag 控制。

如果启用本机默认 ClickHouse（`localhost:9000/default`），可运行：

~~~bash
make -C infra/packages/clickhouse migrate-local
~~~

`migrate-local` 的连接串在上游 Makefile 中是固定值。远端或自定义数据库应显式传入服务连接串，并继续使用官方 `_migrations` tracking 表：

~~~bash
GOOSE_DRIVER=clickhouse \
GOOSE_DBSTRING="$CLICKHOUSE_CONNECTION_STRING" \
go -C infra/packages/clickhouse tool goose \
  -table "_migrations" -dir migrations up
~~~

ClickHouse 可用于指标和事件数据，但不是 API、Client Proxy 或 Orchestrator 的核心启动依赖。

## 部署组合

### 单节点验证

至少保留 PostgreSQL 和 Redis，并显式配置：

~~~bash
SERVICE_DISCOVERY_PROVIDER=local
LOCAL_ORCHESTRATOR_ADDRESS=127.0.0.1:5008
USE_LOCAL_NAMESPACE_STORAGE=true
REDIS_URL=127.0.0.1:6379
LOKI_URL=
VOLUME_TOKEN_ENABLED=false
~~~

`LOKI_URL` 必须存在，但允许显式为空。禁用 Volume Token 后才可省略四个签名变量。

### Nomad 多节点

使用默认 `SERVICE_DISCOVERY_PROVIDER=nomad`，部署上游 jobspec，并为 PostgreSQL、Redis、对象存储和 Nomad/Consul 提供生产级高可用。Orchestrator system job 不会在一个节点故障后“迁移到另一台已运行 system allocation 的节点”；节点恢复或被替换后，Nomad 才在符合条件的节点创建 allocation。

### Kubernetes 多节点

使用 `SERVICE_DISCOVERY_PROVIDER=kubernetes`，确保 Pod label、API RBAC、`hostNetwork`、固定端口和 Service 暴露方式与 discovery 实现一致。Redis、PostgreSQL 和 migrations 的要求不因编排平台改变。

~~~bash
git -C infra rev-parse HEAD
rg -n 'SERVICE_DISCOVERY_PROVIDER|LOCAL_ORCHESTRATOR_ADDRESS' infra/packages/api/internal/cfg/model.go
rg -n 'USE_LOCAL_NAMESPACE_STORAGE|ALLOW_SANDBOX_INTERNAL_CIDRS|SANDBOX_EGRESS_DSCP' infra/packages/orchestrator/pkg/sandbox/network

~~~
