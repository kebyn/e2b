## 1. Auth Provider / Ory

### 当前用途
- API 和 Dashboard API 通过 `AUTH_PROVIDER_CONFIG` 校验外部 JWT/OIDC provider。
- Dashboard API 通过 Ory 管理用户 profile、identity external id 和 OIDC 身份映射。
- Supabase 可以作为外部 IdP 的一种历史/可选实现，但当前代码不再读取旧的 Supabase JWT secret 专用变量。

### 代码依赖分析

```go
// infra/packages/api/internal/cfg/model.go
AuthProvider auth.ProviderConfig `env:"AUTH_PROVIDER_CONFIG"`

// infra/packages/auth/pkg/auth/verifier.go
type ProviderConfig struct {
    JWT []oidc.Config `json:"jwt"`
}

// infra/packages/dashboard-api/internal/cfg/model.go
OrySDKURL          string `env:"ORY_SDK_URL"`
OryProjectAPIToken string `env:"ORY_PROJECT_API_TOKEN,unset"`
OryIssuerURL       string `env:"ORY_ISSUER_URL"`
```

### 多节点 HA 必要性

| 场景 | 是否需要 | 原因 |
|------|----------|------|
| 单节点开发 | ❌ | 可用 API Key 代替 |
| 多节点生产 | ⚠️ | 需要统一用户认证 |
| 私有化部署 | ⚠️ | 使用自有 OIDC/JWT provider；Dashboard 还需 Ory |

### 私有化替换方案

#### 方案 A：配置自有 OIDC/JWT provider（推荐）

```bash
AUTH_PROVIDER_CONFIG='{
  "jwt": [
    {
      "issuer": {
        "url": "https://auth.example.com",
        "audiences": ["e2b-api"],
        "audienceMatchPolicy": "MatchAny"
      },
      "cacheDuration": "30m"
    }
  ]
}'
```

如果启用 Dashboard API：

```bash
ORY_SDK_URL=https://ory.example.com
ORY_PROJECT_API_TOKEN=...
ORY_ISSUER_URL=https://auth.example.com
```

`ORY_ISSUER_URL` 必须与 `AUTH_PROVIDER_CONFIG.jwt[].issuer.url` 中的一个 issuer 匹配；当只有一个 issuer 时 Dashboard API 可从该 issuer 推导。

#### 方案 B：只保留服务端 API Key/Admin Token

不配置 `AUTH_PROVIDER_CONFIG` 时，外部 JWT/OIDC 用户登录不可用，但 API Key、access token、admin token 等路径仍按代码中的认证器工作。此模式适合内部服务调用或不部署 Dashboard 的最小私有化环境。

## 2. Nomad

### 当前用途
1. **节点发现**：API 查询可用的 Orchestrator 节点
2. **任务调度**：管理 Orchestrator/Template Manager 部署
3. **健康检查**：监控节点状态

### 代码依赖分析

```go
// infra/packages/api/internal/cfg/model.go
ServiceDiscoveryProvider string `env:"SERVICE_DISCOVERY_PROVIDER" envDefault:"nomad"`

const (
    ServiceDiscoveryProviderNomad      = "nomad"
    ServiceDiscoveryProviderKubernetes = "kubernetes"
    ServiceDiscoveryProviderLocal      = "local"
)

// infra/packages/api/internal/handlers/store.go
switch config.ServiceDiscoveryProvider {
case cfg.ServiceDiscoveryProviderKubernetes:
    // in-cluster Kubernetes client + Pod label selector
case cfg.ServiceDiscoveryProviderLocal:
    // LOCAL_ORCHESTRATOR_ADDRESS
default:
    // Nomad service discovery + optional legacy node-pool fallback
}
```

### 多节点 HA 必要性

| 场景 | 是否需要 | 原因 |
|------|----------|------|
| 单节点开发 | ❌ | 可用 `SERVICE_DISCOVERY_PROVIDER=local` |
| 多节点生产 | ✅ | 需要动态节点发现和调度；默认路径是 Nomad |
| 私有化部署 | ⚠️ | 可用内置 Kubernetes discovery 替代 Nomad |

### 私有化替换方案

#### 方案 A：使用内置 Kubernetes discovery（推荐）

```bash
SERVICE_DISCOVERY_PROVIDER=kubernetes
K8S_NAMESPACE=e2b
K8S_ORCHESTRATOR_POD_LABEL_SELECTOR=app.kubernetes.io/name=orchestrator
K8S_TEMPLATE_MANAGER_POD_LABEL_SELECTOR=app.kubernetes.io/name=template-manager
```

API Pod 需要能通过 ServiceAccount 访问 K8s API 并 list/watch 当前 namespace 的 Pod。

#### 方案 B：单 Orchestrator 本地/静态模式

```bash
SERVICE_DISCOVERY_PROVIDER=local
LOCAL_ORCHESTRATOR_ADDRESS=10.0.1.10:5008
```

当前 `local` provider 只指向一个 Orchestrator，适合开发/小规模验证，不是多节点 HA discovery。

#### 方案 C：新增 static/Redis discovery（需要改代码）

如果必须支持多节点静态配置文件或 Redis 服务注册，才需要新增 provider。新增值不能复用当前 `SERVICE_DISCOVERY_PROVIDER` 的合法枚举，需同步修改 `infra/packages/api/internal/cfg/model.go` 的校验和 `infra/packages/api/internal/handlers/store.go` 的分支。

**环境变量变更：**
```bash
SERVICE_DISCOVERY_PROVIDER=nomad       # 默认
SERVICE_DISCOVERY_PROVIDER=kubernetes  # K8s
SERVICE_DISCOVERY_PROVIDER=local       # 单节点静态地址
```

## 3. Consul

### 当前用途
- **仅用于网络 Slot 存储**（IP 地址分配的分布式锁）

### 代码依赖分析

```go
// infra/packages/orchestrator/internal/sandbox/network/storage_kv.go
type StorageKV struct {
    consulClient *consulApi.Client  // 用于 IP Slot 分配
}

func NewStorageKV(nodeID string, config Config) (*StorageKV, error) {
    consulToken := utils.RequiredEnv("CONSUL_TOKEN", "...")
    consulClient, err := newConsulClient(consulToken)
    // ...
}
```

### 降级机制（已有）

```go
// infra/packages/orchestrator/main.go
func newStorage(ctx context.Context, nodeID string, config network.Config) (network.Storage, error) {
    // 开发环境或配置为本地时，使用本地存储
    if env.IsDevelopment() || config.UseLocalNamespaceStorage {
        return network.NewStorageLocal(ctx, config)  // 文件系统 + 内存
    }
    // 生产环境使用 Consul KV
    return network.NewStorageKV(nodeID, config)
}
```

### 多节点 HA 必要性

| 场景 | 是否需要 | 原因 |
|------|----------|------|
| 单节点开发 | ❌ | 使用本地存储 |
| 多节点生产 | ⚠️ | 仅当多节点共享 IP 池时需要 |
| 私有化部署 | ⚠️ | 可替换为 Redis |

### 私有化替换方案

#### 方案 A：替换为 Redis（推荐，需要新增代码）

> **重要说明**：当前代码中只有 Consul KV 实现（`storage_kv.go`），Redis 替换方案需要新增代码文件。

```go
// 当前代码状态（infra/packages/orchestrator/pkg/sandbox/network/storage_kv.go）
// 只有 Consul 实现，需要新增 Redis 实现

// 新增: infra/packages/orchestrator/pkg/sandbox/network/storage_redis.go
type StorageRedis struct {
    client redis.UniversalClient
    nodeID string
}

func (s *StorageRedis) Acquire(ctx context.Context) (*Slot, error) {
    // 使用 Redis SETNX 实现分布式锁
    for i := 0; i < 10; i++ {
        slotIdx := rand.Intn(s.slotsSize)
        key := fmt.Sprintf("ip-slot:%s:%d", s.nodeID, slotIdx)

        ok, err := s.client.SetNX(ctx, key, "1", 0).Result()
        if err == nil && ok {
            return NewSlot(key, slotIdx, s.config)
        }
    }
    return nil, fmt.Errorf("no slots available")
}
```

**代码修改步骤**：
1. 新增 `storage_redis.go` 实现 `Storage` 接口
2. 修改 `newStorage()` 函数，增加 Redis 选项判断
3. 添加环境变量 `IP_SLOT_STORAGE=redis`

#### 方案 B：每节点独立 IP 池（无需修改代码）

> **推荐方案**：无需修改代码，只需正确配置即可。

```bash
# 节点 1: IP 池 10.0.1.0/24
# 节点 2: IP 池 10.0.2.0/24
# 节点 3: IP 池 10.0.3.0/24

# 环境变量
USE_LOCAL_NAMESPACE_STORAGE=true  # 使用本地存储
```

**工作原理**：
- 每个节点使用独立的 IP 地址池
- 无需跨节点协调，本地文件系统存储即可
- 适用于大多数私有化部署场景

**环境变量变更：**
```bash
# 原有
CONSUL_TOKEN=xxx

# 替换为
IP_SLOT_STORAGE=redis  # 或 local
REDIS_URL=localhost:6379
```

## 4. LaunchDarkly

### 当前用途
- **功能开关**：控制 40+ 个运行时参数
- **动态配置**：无需重启调整性能参数
- **灰度发布**：按 team/template 控制功能

> Feature Flags 的完整列表、默认值、YAML / Unleash 替代方案请看：[`FeatureFlags私有化部署方案.md`](../feature-flags/FeatureFlags私有化部署方案.md)
>
> 本节只回答：**私有化时是否必须保留 LaunchDarkly**。

### 代码依赖分析

```go
// infra/packages/shared/pkg/featureflags/client.go
func NewClient() (*Client, error) {
    if launchDarklyApiKey == "" {
        // 离线模式：使用代码中的默认值
        return NewClientWithDatasource(launchDarklyOfflineStore)
    }
    // 在线模式：连接 LaunchDarkly
    ldClient, err := ldclient.MakeClient(launchDarklyApiKey, waitForInit)
    return &Client{ld: ldClient}, nil
}
```

### 降级机制（已内置）

- 所有 flag 都有 fallback 默认值
- 未配置 `LAUNCH_DARKLY_API_KEY` 时自动进入离线模式
- 因此 LaunchDarkly 在私有化部署中通常不是阻塞组件

### 多节点 HA 必要性

| 场景 | 是否需要 | 原因 |
|------|----------|------|
| 单节点开发 | ❌ | 使用默认值 |
| 多节点生产 | ❌ | 使用默认值即可 |
| 私有化部署 | ❌ | 无需修改，使用默认值 |

### 私有化方案

**默认建议：无需任何修改。** 代码已内置离线默认值。

如果你仍然需要动态配置或灰度能力，请不要在本节重复维护方案，统一看：

- [`FeatureFlags私有化部署方案.md`](../feature-flags/FeatureFlags私有化部署方案.md)

## 5. Redis

### 当前用途
1. **Sandbox Catalog**：路由表（sandbox → orchestrator 映射）
2. **状态存储**：sandbox 生命周期状态
3. **模板缓存**：减少数据库查询
4. **事件流**：异步事件处理
5. **P2P 注册表**：节点发现
6. **速率限制**：API 限流

### 降级机制（已内置）

```go
// infra/packages/client-proxy/main.go
if redisClient != nil {
    catalog = NewRedisSandboxesCatalog(redisClient, featureFlagsClient)
} else {
    catalog = NewMemorySandboxesCatalog()  // 内存降级
}

// infra/packages/api/internal/orchestrator/orchestrator.go
if redisClient != nil {
    routingCatalog = e2bcatalog.NewRedisSandboxesCatalog(redisClient, featureFlags)
} else {
    routingCatalog = e2bcatalog.NewMemorySandboxesCatalog()
}
```

### 多节点 HA 必要性

| 场景 | 是否需要 | 原因 |
|------|----------|------|
| 单节点开发 | ❌ | 内存降级 |
| 多节点生产 | ✅ | 跨节点共享状态 |
| 私有化部署 | ✅ | 必须部署 Redis |

### 私有化部署建议

**无需修改代码**，只需部署 Redis：

```bash
# 单节点 Redis（简单部署）
docker run -d --name redis redis:7-alpine

# 环境变量
REDIS_URL=redis-host:6379

# 高可用：Redis 主从前放 HAProxy/VIP，应用仍只连接一个 host:port
REDIS_URL=redis-vip:6379

# 或 Redis Cluster 模式，逗号分隔多节点
REDIS_CLUSTER_URL=redis-node-1:6379,redis-node-2:6379,redis-node-3:6379
```

应用侧不支持 Sentinel/Failover 客户端协议；如果使用 Sentinel，只能把它作为 Redis 主从选主和运维查询机制，不能把 Sentinel 端口配置给 `REDIS_URL`。

## 6. ClickHouse

### 当前用途
- **指标存储**：sandbox/host 性能指标
- **事件日志**：sandbox 事件审计

### 降级机制（已内置）

```go
// infra/packages/api/internal/handlers/store.go
if clickhouseConnectionString == "" {
    clickhouseStore = clickhouse.NewNoopClient()  // 空操作
} else {
    clickhouseStore, err = clickhouse.New(clickhouseConnectionString)
}
```

### 多节点 HA 必要性

| 场景 | 是否需要 | 原因 |
|------|----------|------|
| 单节点开发 | ❌ | Noop 降级 |
| 多节点生产 | ⚠️ | 推荐，用于监控 |
| 私有化部署 | ⚠️ | 可选，不影响核心功能 |

### 私有化部署建议

**无需修改代码**，可选部署：

```bash
# 单节点 ClickHouse
docker run -d --name clickhouse clickhouse/clickhouse-server:25-alpine

# 环境变量（可选）
CLICKHOUSE_CONNECTION_STRING=clickhouse://clickhouse:clickhouse@clickhouse-host:9000/default
```

## 代码修改优先级

### 默认无需改代码（多节点部署）

| 组件 | 修改内容 | 工作量 |
|------|----------|--------|
| **Service Discovery** | 在 `nomad` / `kubernetes` / `local` 中选择 | 配置 |
| **Redis** | 无需修改，直接部署 | 无 |

### 需要按环境配置

| 组件 | 修改内容 | 工作量 |
|------|----------|--------|
| **Auth Provider / Ory** | 配置 `AUTH_PROVIDER_CONFIG` 与 Ory env | 配置 |
| **Consul** | 替换为 Redis 或本地存储 | 小 |

### 无需修改

| 组件 | 原因 |
|------|------|
| **LaunchDarkly** | 已内置离线模式 |
| **ClickHouse** | 已内置 Noop 降级 |
