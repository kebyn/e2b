## 1. 认证 Provider / OIDC / Ory

### 是什么

`2026.28` 的认证入口已经从 Supabase-only JWT secret 转为 provider-based 配置。API 和 Dashboard API 都通过 `AUTH_PROVIDER_CONFIG` 接收外部 JWT/OIDC issuer 配置；Dashboard API 另外使用 Ory 管理用户资料、外部身份和 profile 数据。

Supabase 仍可作为外部 OIDC/JWT provider 的一种实现，但不再是当前代码中的专用配置项。

### 在 E2B 中的用途

```
用户/服务请求 → 外部 OIDC/JWT Provider 签发 Token → E2B API/Dashboard API 验证 → 授权访问
                                      │
                                      └── Dashboard API 通过 Ory 管理用户资料/身份映射
```

1. **API Key / Admin Token**：保留 `X-API-Key`、admin token 等服务端认证路径。
2. **外部 JWT/OIDC**：`AUTH_PROVIDER_CONFIG.jwt[]` 声明 issuer、audience 策略和 JWKS discovery。
3. **Dashboard Profile**：Dashboard API 需要 `ORY_SDK_URL`、`ORY_PROJECT_API_TOKEN`、`ORY_ISSUER_URL`，并要求 Ory issuer 与 `AUTH_PROVIDER_CONFIG.jwt[].issuer.url` 匹配或可从单一 issuer 推导。
4. **用户识别**：认证层从 JWT claims / Ory identity 中解析用户身份，关联到 E2B 的 team、sandbox、template 等资源。

### 工作流程

```
┌─────────────┐     ┌────────────────┐     ┌────────────────────┐
│   Browser   │────▶│ OIDC/JWT IdP   │────▶│ E2B API / Dashboard │
│  / Client   │     │ Ory / Hydra 等 │     │ Auth Provider Verify│
└─────────────┘     └────────────────┘     └────────────────────┘
      │                       │                       │
      │ 1. Login / Client Cred│                       │
      │──────────────────────▶│                       │
      │                       │ 2. JWT Token          │
      │◀──────────────────────│                       │
      │ 3. Request + Bearer JWT                        │
      │───────────────────────────────────────────────▶│
      │                       │ 4. OIDC discovery/JWKS │
      │                       │◀──────────────────────▶│
      │ 5. Response                                    │
      │◀───────────────────────────────────────────────│
```

### 环境变量

| 变量 | 说明 |
|------|------|
| `AUTH_PROVIDER_CONFIG` | 外部 JWT/OIDC provider 配置，JSON 结构为 `{"jwt":[...]}` |
| `CLIENT_PROXY_OIDC_ISSUER_URL` | API Edge gRPC 用于校验 Client Proxy token 的 issuer URL；Edge 通道可用时必需 |
| `ORY_SDK_URL` | Dashboard API 访问 Ory 的 SDK/API URL |
| `ORY_PROJECT_API_TOKEN` | Dashboard API 访问 Ory 的 Project API Token |
| `ORY_ISSUER_URL` | Dashboard/Ory 对应的 OIDC issuer URL |

### `AUTH_PROVIDER_CONFIG` 示例

```json
{
  "jwt": [
    {
      "issuer": {
        "url": "https://auth.example.com",
        "audiences": ["e2b-api"],
        "audienceMatchPolicy": "MatchAny"
      },
      "cacheDuration": "30m"
    }
  ]
}
```

### 支持的认证方式

在 `infra/packages/auth/pkg/auth/` 和 API/Dashboard API 路由中组合使用：

```go
auth.NewApiKeyAuthenticator(...)              // X-API-Key header
auth.NewAccessTokenAuthenticator(...)         // Bearer access token
auth.NewAuthProviderBearerAuthenticator(...)  // Bearer JWT/OIDC provider
auth.NewAuthProviderTeamAuthenticator(...)    // JWT/OIDC provider + team ID
auth.NewAdminApiKeyAuthenticator(...)         // 管理员 token
```

## 2. Nomad

### 是什么
HashiCorp Nomad 是一个工作负载编排器，类似 Kubernetes。E2B 使用 Nomad 来**管理和调度 Orchestrator 节点**。

### 在 E2B 中的用途

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   E2B API   │────▶│    Nomad    │────▶│ Orchestrator│
│             │     │  (Cluster)  │     │   Nodes     │
└─────────────┘     └─────────────┘     └─────────────┘
      │                    │                    │
      │ 1. Query nodes     │                    │
      │───────────────────▶│                    │
      │                    │                    │
      │ 2. Node list       │                    │
      │◀───────────────────│                    │
      │                    │                    │
      │ 3. gRPC call to specific node          │
      │─────────────────────────────────────────▶
```

### 主要功能

1. **节点发现**：API 通过 Nomad 查询可用的 Orchestrator 节点
2. **工作分配**：Nomad 管理 Orchestrator 和 Template Manager 的部署
3. **健康检查**：Nomad 监控节点健康状态
4. **任务生命周期**：按 job 定义维护 allocation；system job 会在符合条件的每个节点运行，故障节点恢复或被替换后再创建对应 allocation

### 代码入口

`infra/packages/api/internal/orchestrator/discovery/nomad.go` 通过 Nomad native service registration 列举节点；`nomad_node_pool.go` 提供由 `NOMAD_ORCHESTRATOR_LEGACY_DISCOVERY_ENABLED` 控制的旧 node-pool 回退。发现后的候选节点还会经过 Orchestrator gRPC `ServiceInfo` 健康探测。

### 环境变量

| 变量 | 必需 | 默认值 | 说明 |
|------|------|--------|------|
| `SERVICE_DISCOVERY_PROVIDER` | ❌ | `nomad` | 可选 `nomad`、`kubernetes`、`local` |
| `NOMAD_ADDRESS` | ❌ | `http://localhost:4646` | Nomad API 地址 |
| `NOMAD_TOKEN` | ❌ | - | Nomad ACL Token（生产环境需要） |
| `NOMAD_ORCHESTRATOR_SERVICE_NAMES` | ❌ | `orchestrator` | 逗号分隔的 Nomad native service 名称 |
| `NOMAD_ORCHESTRATOR_LEGACY_DISCOVERY_ENABLED` | ❌ | `true` | 是否合并旧 node-pool 回退结果 |

### 本地开发模式

在本地开发时，可以跳过 Nomad：
- 显式设置 `SERVICE_DISCOVERY_PROVIDER=local`
- 用 `LOCAL_ORCHESTRATOR_ADDRESS` 指向单个 Orchestrator，默认 `127.0.0.1:5008`
- 适用于单节点测试

仅省略 `NOMAD_ADDRESS` 不会切换 provider；API 仍使用默认 `nomad` 并访问 `http://localhost:4646`。

## 3. LaunchDarkly (Feature Flags)

### 是什么
LaunchDarkly 是一个功能开关（Feature Flags）服务，允许在**不重新部署代码**的情况下动态控制应用行为。

### 在 E2B 中的用途

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  LaunchDarkly│     │  E2B API /  │     │  Runtime    │
│  (Dashboard) │────▶│  Orchestrator│────▶│  Behavior   │
└─────────────┘     └─────────────┘     └─────────────┘
      │                    │                    │
      │ 1. Set flag        │                    │
      │  max-sandboxes=50  │                    │
      │                    │                    │
      │                    │ 2. Read flag value │
      │                    │───────────────────▶│
      │                    │                    │
      │                    │ 3. Apply limit: max 50 sandboxes
```

### 在 E2B 中主要控制什么

LaunchDarkly 在这里主要承担三类职责：

1. **性能调优**：例如每节点 sandbox 上限、GCS 并发、NBD 连接数、prefetch worker 数
2. **功能开关**：例如持久卷、P2P 传输、网络 transform、按 label 调度
3. **动态配置**：例如 Firecracker 版本映射、限流配置、出口流量限制、chunker 配置

### 为什么它和普通环境变量不同

- 环境变量更适合“启动时确定”的静态配置
- LaunchDarkly 更适合“运行时可调整”的策略和开关
- 它支持按 **team / template / sandbox** 维度做差异化控制

> 完整 Flag 列表、精确默认值、YAML/Unleash 替代方案，请看：[`FeatureFlags私有化部署方案.md`](../feature-flags/FeatureFlags私有化部署方案.md)

#### 6. 按上下文的目标控制

```go
// 支持按 team、sandbox、template 等维度进行差异化配置
ctx = featureflags.AddToContext(ctx,
    featureflags.TeamContext(teamID),              // 按团队
    featureflags.SandboxContext(sandboxID),         // 按 sandbox
    featureflags.TemplateContext(templateID),       // 按模板
)
```

### 使用示例

```go
// 读取布尔标志
if flags.BoolFlag(ctx, featureflags.PeerToPeerChunkTransferFlag) {
    // 启用 P2P 传输
}

// 读取整数标志
maxSandboxes := flags.IntFlag(ctx, featureflags.MaxSandboxesPerNode)

// 读取 JSON 标志
versions := flags.JSONFlag(ctx, featureflags.FirecrackerVersions)
```

### 环境变量

| 变量 | 必需 | 说明 |
|------|------|------|
| `LAUNCH_DARKLY_API_KEY` | ❌ | LaunchDarkly SDK Key。未设置时使用默认值（离线模式） |

> 完整配置参考请看：[`启动参数详解.md`](../reference/启动参数详解.md)；完整 Feature Flags 方案请看：[`FeatureFlags私有化部署方案.md`](../feature-flags/FeatureFlags私有化部署方案.md)

### 离线模式

当未配置 `LAUNCH_DARKLY_API_KEY` 时：
- 使用代码中的 `fallback` 默认值
- 适用于本地开发和测试
- 不支持动态更新

## 4. Redis

### 是什么
Redis 是一个高性能的内存键值存储。E2B 使用 Redis 作为**分布式缓存和状态存储**。

### 在 E2B 中的用途

```
┌─────────────────────────────────────────────────────────────┐
│                         Redis                               │
├─────────────────────────────────────────────────────────────┤
│  Sandbox Catalog       →  Sandbox 位置和状态（路由用）      │
│  API caches/indexes    →  模板、快照、过期索引和限流状态    │
│  Sandbox events        →  Orchestrator 事件流（Streams）    │
│  Peer registry         →  Orchestrator P2P 节点注册表       │
└─────────────────────────────────────────────────────────────┘
```

### 主要数据结构

#### 1. Sandbox Catalog（路由表）

```go
// infra/packages/shared/pkg/sandbox-catalog/catalog_redis.go
// API 写入、Client Proxy 查询：sandbox:catalog:<sandboxID>
catalog := sandbox_catalog.NewRedisSandboxCatalog(redisClient)
```

#### 2. Sandbox 状态存储

```go
// API 的 expiration index、限流、模板和快照缓存分别由当前实现维护。
// key/value 是内部契约，不应按旧版本示例手工构造。
```

#### 3. 模板/快照缓存

```go
// API 使用通用 Redis cache 包缓存模板和快照元数据
cache.NewRedisCache[*TemplateInfo](RedisConfig[*TemplateInfo]{...})
```

#### 4. 事件流（Redis Streams）

```go
// 用于异步事件处理
event.NewRedisStreamsDelivery[SandboxEvent](redisClient, "sandbox-events")
```

#### 5. P2P 节点注册表

```go
// 用于 Orchestrator 之间发现彼此
peerclient.NewRedisRegistry(redisClient, nodeAddress)
```

### 架构图

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ Client Proxy│────▶│    Redis    │◀────│ Orchestrator│
│             │     │  (Catalog)  │     │   Node A    │
└─────────────┘     └─────────────┘     └─────────────┘
      │                    ▲                    │
      │ Query sandbox      │                    │
      │ location           │ Register           │
      │                    │ sandbox            │
      ▼                    │                    ▼
┌─────────────┐            │            ┌─────────────┐
│   Sandbox   │            │            │   Sandbox   │
│   (HTTP)    │◀───────────┼───────────▶│   (gRPC)    │
└─────────────┘            │            └─────────────┘
                           │
                    ┌─────────────┐
                    │  Orchestrator│
                    │   Node B    │
                    └─────────────┘
```

### 依赖行为

API 和 Client Proxy 都在启动路径中直接调用 `factories.NewRedisClient`，错误会终止启动；当前不存在内存 catalog。Orchestrator 单独允许 `ErrRedisDisabled`，此时不注册 Redis peer registry，也不创建 Redis sandbox event delivery。

### 环境变量

| 变量 | 必需 | 说明 |
|------|------|------|
| `REDIS_URL` | 条件 | 单节点地址，格式：`host:port` |
| `REDIS_CLUSTER_URL` | 条件 | 单个 Redis Cluster 引导端点，优先于 `REDIS_URL` |
| `REDIS_TLS_CA_BASE64` | ❌ | TLS CA 证书（base64 编码） |
| `REDIS_POOL_SIZE` | ❌ | 连接池大小（API `160`、Client Proxy `40`、Orchestrator `5`） |
| `REDIS_MIN_IDLE_CONNS` | ❌ | 仅 Orchestrator 读取，默认 `2` |

API 和 Client Proxy 必须设置 `REDIS_URL` 或 `REDIS_CLUSTER_URL` 之一。`REDIS_CLUSTER_URL` 不是逗号分隔列表：源码将整个值作为 `redis.ClusterOptions.Addrs` 的单个元素。应用不实现 Redis Sentinel/Failover 客户端；使用 Sentinel 管理主从时，应在应用前提供 HAProxy、VIP 或云负载均衡器，并继续给 `REDIS_URL` 一个 `host:port`。

### 连接模式选择

```go
// infra/packages/shared/pkg/factories/redis.go
switch {
case config.RedisClusterURL != "":
    // Redis Cluster（GCP Memorystore 推荐）
    redisClient = redis.NewClusterClient(clusterOpts)
case config.RedisURL != "":
    // 单节点 Redis
    redisClient = redis.NewClient(opts)
default:
    // Redis 禁用
    return nil, ErrRedisDisabled
}
```

## 5. Volume Token

### 是什么
Volume Token 是一个 JWT（JSON Web Token），用于**授权访问持久卷**。它类似于 AWS S3 的 Presigned URL，但用于 E2B 的持久卷系统。

### 在 E2B 中的用途

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Client    │────▶│   E2B API   │────▶│  Orchestrator│
│             │     │ (Issue JWT) │     │ (Verify JWT) │
└─────────────┘     └─────────────┘     └─────────────┘
      │                    │                    │
      │ 1. Create volume   │                    │
      │───────────────────▶│                    │
      │                    │                    │
      │ 2. Volume Token (JWT)                  │
      │◀───────────────────│                    │
      │                    │                    │
      │ 3. Mount volume + Token                │
      │─────────────────────────────────────────▶
      │                    │                    │
      │                    │  4. Verify token   │
      │                    │◀───────────────────│
      │                    │                    │
      │ 5. Access granted  │                    │
      │◀─────────────────────────────────────────│
```

### Token 结构

JWT header 包含签名算法和当前密钥名称：

```json
{
  "alg": "ES256",
  "typ": "JWT",
  "tokid": "local-dev-2026-03-03"
}
```

claims 包含标准 JWT 字段以及团队和卷标识：

```json
{
  "aud": "cluster-id",
  "exp": 1711334400,
  "iat": 1711330800,
  "iss": "local.e2b.dev",
  "jti": "unique-token-id",
  "nbf": 1711330800,
  "sub": "team-uuid",
  "teamid": "team-uuid",
  "volid": "volume-uuid",
  "voltype": "persistent"
}
```

### 签发流程

```go
// infra/packages/api/internal/handlers/volume_token.go
func generateVolumeContentToken(config cfg.VolumesTokenConfig, volume queries.Volume, team *types.Team) (string, error) {
    claims := jwt.MapClaims{
        "aud":     clusterID.String(),
        "exp":     jwt.NewNumericDate(expiration),
        "iat":     jwt.NewNumericDate(now),
        "iss":     config.Issuer,
        "jti":     uuid.NewString(),
        "nbf":     jwt.NewNumericDate(now),
        "sub":     team.ID.String(),
        "teamid":  team.ID.String(),
        "volid":   volume.ID.String(),
        "voltype": volume.VolumeType,
    }

    token := jwt.NewWithClaims(config.SigningMethod, claims)
    token.Header["tokid"] = config.SigningKeyName
    return token.SignedString(config.SigningKey)
}
```

### 密钥类型说明

| 类型 | 算法 | 密钥格式 | 适用场景 |
|------|------|----------|----------|
| `ECDSA` | ES256 (P-256) | PEM 私钥 | **推荐**：体积小、性能好 |
| `RS256` | RSA 2048+ | PEM 私钥 | 兼容性好，体积大 |
| `HMAC` | HS256 | 任意字节 | 简单，需安全分发 |
| `EdDSA` | Ed25519 | PEM 私钥 | 最新最快，部分系统不支持 |

### 密钥生成示例

```bash
# ECDSA（推荐）
openssl ecparam -name prime256v1 -genkey -noout -out private.pem
openssl ec -in private.pem -pubout -out public.pem

# RSA
openssl genrsa -out private.pem 2048
openssl rsa -in private.pem -pubout -out public.pem

# 配置环境变量
VOLUME_TOKEN_SIGNING_KEY="ECDSA:$(base64 -w0 private.pem)"
```

### 环境变量

| 变量 | 必需 | 默认值 | 说明 |
|------|------|--------|------|
| `VOLUME_TOKEN_ENABLED` | ❌ | `true` | 是否启用卷内容 token 签名 |
| `VOLUME_TOKEN_ISSUER` | 条件 | - | 启用时必需；签发者标识，如 `e2b.dev` |
| `VOLUME_TOKEN_SIGNING_METHOD` | 条件 | - | 启用时必需；JWT 签名算法 |
| `VOLUME_TOKEN_SIGNING_KEY` | 条件 | - | 启用时必需；签名密钥，格式：`TYPE:base64(PEM)` |
| `VOLUME_TOKEN_SIGNING_KEY_NAME` | 条件 | - | 启用时必需；写入 JWT `tokid` header 的密钥名称 |
| `VOLUME_TOKEN_DURATION` | ❌ | `1h` | Token 有效期 |

### 密钥轮换流程

```bash
# 1. 生成新密钥
openssl ecparam -name prime256v1 -genkey -noout -out new-key.pem

# 2. 更新当前签名配置
VOLUME_TOKEN_SIGNING_KEY="ECDSA:$(base64 -w0 new-key.pem)"
VOLUME_TOKEN_SIGNING_KEY_NAME="prod-2026-03-25"

# 3. 与 token 验证方协调上线并等待旧 token 过期
```

当前 API 配置只接受一把签名密钥和一个名称，没有旧密钥 keyring 或并行验证配置。轮换时必须与实际 token 验证方协调，不能假设仅在 API 中“保留旧密钥”即可兼容旧 token。

### 安全考虑

1. **密钥存储**：生产环境应使用 GCP Secrets Manager 或类似服务
2. **有效期控制**：`VOLUME_TOKEN_DURATION` 应根据使用场景调整
3. **集群隔离**：`aud` 字段验证 token 只在特定集群有效
4. **团队隔离**：`teamid` 字段确保 token 只访问团队自己的卷

## 6. 制品仓库（Artifacts Registry）

### 是什么
制品仓库用于存储模板构建后的 Docker 镜像，供 Orchestrator 和 Template Manager 拉取使用。

### 在 E2B 中的用途

```
┌─────────────┐     ┌─────────────┐     ┌─────────────────────┐
│Template     │────▶│  Build      │────▶│ Artifacts Registry  │
│Manager      │     │  Process    │     │ (GCP/AWS/Local)     │
└─────────────┘     └─────────────┘     └─────────────────────┘
       │                                        │
       │ 拉取镜像                               │
       └────────────────────────────────────────┘
```

### 支持的提供者

| 提供者 | 环境变量值 | 说明 |
|--------|-----------|------|

## 7. Sandbox 网络分配

### 是什么
E2B 为每个 Sandbox 分配独立的 IP 地址，用于 host 和 VM 内部通信。

### 网络架构

```
┌─────────────────────────────────────────────────────────────┐
│                    Sandbox 网络                               │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  Host Network CIDR: 10.11.0.0/16                            │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ Slot 1: 10.11.0.1/32  (Host IP - host 访问 sandbox) │    │
│  │ Slot 2: 10.11.0.2/32                                 │    │
│  │ ...                                                  │    │
│  │ Slot N: 10.11.0.N/32  (支持 ~65534 个 sandbox)      │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
│  Vrt Network CIDR: 10.12.0.0/16                             │
│  ┌─────────────────────────────────────────────────────┐    │
│  │ Slot 1: 10.12.0.0/31  (vpeer + veth, 2 IPs)         │    │
│  │ Slot 2: 10.12.0.2/31                                 │    │
│  │ ...                                                  │    │
│  │ Slot N: 10.12.0.(2N)/31                              │    │
│  └─────────────────────────────────────────────────────┘    │
│                                                             │
└─────────────────────────────────────────────────────────────┘
```

### 环境变量

| 变量 | 默认值 | 说明 |
|------|--------|------|
| `SANDBOXES_HOST_NETWORK_CIDR` | `10.11.0.0/16` | Host IP 网络 |
| `SANDBOXES_VRT_NETWORK_CIDR` | `10.12.0.0/16` | Vrt（vpeer/veth）网络 |
| `USE_LOCAL_NAMESPACE_STORAGE` | `false` | 使用本机 namespace 状态而非 Consul KV |
| `ALLOW_SANDBOX_INTERNAL_CIDRS` | 空 | 逗号分隔的内部 CIDR 白名单 |
| `SANDBOX_EGRESS_DSCP` | `0` | 出站 DSCP 标记，范围 `0..63`；`0` 表示不标记 |

### IP 分配机制
- **Host IP**：每个 slot 分配一个 `/32` IP，用于 host 直接访问 sandbox
- **Vrt IP**：每个 slot 分配一个 `/31` 子网（2 IPs），用于 sandbox 内部 vpeer 和 veth 通信
- **存储方式**：
  - `USE_LOCAL_NAMESPACE_STORAGE=false`：通过 Consul KV 保存 slot reservation（需要 `CONSUL_TOKEN`）；key 包含当前 `nodeID`
  - `USE_LOCAL_NAMESPACE_STORAGE=true`：根据本机 namespace 状态进行分配，不跨节点共享状态

## 8. Client Proxy 请求寻址

### 两种寻址方式

Client Proxy 与 Orchestrator 复用 `infra/packages/shared/pkg/proxy/host.go` 的解析器：

```text
{port}-{sandboxID}.<DOMAIN_NAME>  -> Host 解析
sandbox.<DOMAIN_NAME>             -> Header 解析（需要两个 routing Header）
localhost / IP + routing Header   -> Header 解析
```

Header 名称为 `E2b-Sandbox-Id` 和 `E2b-Sandbox-Port`。共享 Host、localhost 或 IP 入口上只要出现任一 Header 就进入 Header 解析；两个 Header 缺一、端口不可解析或 Sandbox ID 无效时请求失败。普通 `{port}-{sandboxID}` Host 始终优先，携带的 Header 不会覆盖它。

IP Host 无法编码 Sandbox ID 和端口，因此 bridge、LAN、on-prem 或直接以节点 IP 暴露 Client Proxy 时，必须使用 Header 寻址。负载均衡器需要显式接收 `sandbox.<DOMAIN_NAME>`，保留原始 Host 和这两个 Header，并把流量转发到 Client Proxy。

### 共享 Host 的兼容转发

当 `orch-accepts-combined-host=false`（默认）时，Client Proxy 在转发到旧式 Orchestrator 解析器前，将共享 Host 改写为 `{port}-{sandboxID}.<DOMAIN_NAME>`；启用该 Flag 后才把共享 Host 和 Header 组合直接交给 Orchestrator。该 Flag 只控制下游兼容改写，不是 Header 寻址的总开关。

### 暂停 Sandbox 的流量触发恢复

Client Proxy 查询 Redis catalog 未命中时，如果 Sandbox 创建配置启用了 `autoResume`，且 Client Proxy 配置了 API internal gRPC 或 Edge gRPC 恢复通道，会通过 API 尝试恢复暂停的 Sandbox，再把请求转发到恢复后的 Orchestrator。请求端的 traffic access token 和 secure Sandbox 的 envd access token 会随恢复调用转发；没有 API gRPC 地址时不会自动发现恢复通道。`sandbox-auto-resume` 已在 `2026.28` 删除，不应作为当前 Feature Flag 配置。

Client Proxy 优先使用 `API_INTERNAL_GRPC_ADDRESS` 的非 TLS 通道，只有该值为空时才选择 `API_EDGE_GRPC_ADDRESS` 并强制 TLS。API 的 `API_EDGE_GRPC_PORT=5109` listener 自身是明文 gRPC，外部 LB/代理必须在前面终止 TLS，不能让 Client Proxy 直接把裸 `api-lb:5109` 当作 Edge TLS endpoint。

Edge handler 始终执行 OIDC 校验：API 必须配置 `CLIENT_PROXY_OIDC_ISSUER_URL`，Client Proxy 必须同时配置三个 `API_EDGE_GRPC_OAUTH_*` 字段，并取得含 `sandboxes:lifecycle` scope、`org_id` 与目标集群组织匹配的 token。API issuer 为空时 Edge 请求会被拒绝；选择 Edge 通道时 OAuth 三字段部分配置会导致 Client Proxy 启动失败，全部为空也只是生成无认证客户端，不能作为可用的匿名 Edge 通道。
