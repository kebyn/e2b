## 1. 核心结论

| 组件 | 是否必须 | 降级方案 | 影响 |
|------|----------|----------|------|
| **Service Discovery** | ✅ **必须** | `nomad` / `kubernetes` / `local` | 多节点 Orchestrator 发现依赖动态发现；Nomad 是默认 HA 路径，K8s 已内置，`local` 仅适合单节点 |
| **Redis** | ✅ **必须** | 无内存降级 | API 与 Client Proxy 无 Redis 会启动失败 |
| **PostgreSQL** | ✅ **必须** | 无降级 | 核心数据存储 |
| **Consul** | ❌ 可选 | 本机 namespace storage | 每台 Orchestrator 独占宿主机网络命名空间时可省略 |
| **LaunchDarkly** | ❌ 可选 | 默认值 | 功能开关不可动态调整 |
| **ClickHouse** | ❌ 可选 | Noop | 无监控指标 |
| **Auth Provider / Ory** | ⚠️ 按需 | API Key/Admin Token | 不部署会缺少外部 JWT/OIDC 用户认证和 Dashboard 用户资料能力 |

## 2. 高可用拓扑总览

```text
用户流量
  -> LB / Nginx / HAProxy
  -> API (3 实例) + Client Proxy (2+ 实例)
  -> SERVICE_DISCOVERY_PROVIDER 发现多个 Orchestrator 节点
  -> Redis 主从 (HAProxy/VIP) / PostgreSQL 主从提供共享状态与持久化
```

### 推荐最小实例数

| 组件 | 最低建议 | 说明 |
|------|----------|------|
| Nomad Server | 3 | 保证仲裁 |
| Redis | 1 主 + 1 从 + HAProxy/VIP，或 Redis Cluster | 应用不支持 Sentinel 协议；单节点模式经 HAProxy/Keepalived 暴露 `REDIS_URL=host:port`，Cluster 模式给 `REDIS_CLUSTER_URL` 单个引导端点 |
| PostgreSQL | 1 主 + 1 从 | 至少具备切换能力 |
| API | 3 | LB 后无状态扩缩容 |
| Client Proxy | 2+ | 保障 sandbox 访问入口 |
| Orchestrator | N 台计算节点 | 由 Nomad system job 管理 |

## 4. 高可用关注点

### 4.1 API 层

- API 必须多实例部署在负载均衡后
- 多实例必须指向同一组 PostgreSQL、Redis 配置；旧版 sandbox storage backend 部署开关已从 2026.28 API config 移除
- 健康检查必须接入 LB 剔除机制
- 启用 Volume Token 时所有实例的 `VOLUME_TOKEN_*`、数据库、Redis 配置必须一致；不用卷时统一设 `VOLUME_TOKEN_ENABLED=false`

### 4.2 Client Proxy 层

- 至少 2 个实例
- 保证 wildcard sandbox 域名入口不单点
- Proxy 的 Redis 连接必须指向共享状态存储

### 4.3 Orchestrator 层

- 通过 Nomad system job 运行在多个计算节点
- system job 在每个符合条件的节点运行；故障节点恢复或替换后才会创建对应 allocation，不会把它迁移到另一个已运行 allocation 的节点
- 可在每台独占宿主机网络命名空间的节点使用 local namespace storage；CIDR 只需避免与本机、Pod 网络和现有路由冲突
- 节点资源（NBD、hugepages、磁盘）要按峰值容量预留

### 4.4 数据层

- **Redis**：源码按配置分别使用 `redis.NewClient` 或 `redis.NewClusterClient`，**不支持 Sentinel/Failover 协议**。主从模式需通过 HAProxy + Keepalived（或 Cloud LB + Health Check）对外暴露单一 VIP；Cluster 模式只配置一个引导端点
- PostgreSQL 至少准备一个可提升的副本
- 必须做主从切换演练，而不是只看“配置存在”

## 5. 高可用验证

### 5.1 健康检查

```bash
# API 健康检查
curl http://api-1:3000/health
curl http://api-2:3000/health
curl http://api-3:3000/health

# Client Proxy 健康检查
curl http://proxy-1:3003/health
curl http://proxy-2:3003/health

# Orchestrator 健康检查（或通过 Nomad）
nomad status e2b-orchestrator
nomad node status
```

### 5.2 Nomad 状态检查

```bash
# 查看 Nomad 集群状态
nomad server members

# 查看 Orchestrator 任务状态
nomad status e2b-orchestrator

# 查看节点分配
nomad node status -verbose
```

### 5.3 Redis 检查

```bash
# Redis 主从状态
redis-cli -h redis-master INFO replication

# HAProxy/VIP 健康（e2b 不支持 Sentinel 协议，使用外层 LB 切主）
redis-cli -h redis-vip -p 6379 INFO replication | grep role
```

### 5.4 PostgreSQL 检查

```bash
# 主库可用性
psql -h postgres -U e2b -c "SELECT 1;"

# 查看复制状态（主库执行）
psql -h postgres -U postgres -c "SELECT * FROM pg_stat_replication;"
```

### 5.5 业务验证

```bash
# 1. 创建 sandbox
# 2. 访问 sandbox URL
# 3. 重启一个 API 实例后再次访问
# 4. 验证会话与路由未丢失
```

## 6. 故障恢复

### 6.1 API 节点故障

```bash
# 自动: 负载均衡器检测健康检查失败后自动剔除
# 手动: 重启服务
systemctl restart e2b-api
```

### 6.2 Orchestrator 节点故障

```bash
# system job 不会迁移到已运行同一 allocation 的其他节点；恢复故障节点，
# 或加入符合约束的新节点，让 Nomad 在该节点创建 allocation
nomad job restart e2b-orchestrator
```

### 6.3 Redis 故障

```bash
# e2b 不支持 Sentinel，需依赖 HAProxy/Keepalived/云 LB 自动剔除故障主
# 验证当前 VIP 指向的角色
redis-cli -h redis-vip -p 6379 INFO replication | grep role

# 手动切换从库为新主
redis-cli -h redis-replica REPLICAOF NO ONE
# 然后更新 HAProxy backend 或 VIP 指向新主
```

### 6.4 PostgreSQL 故障

```bash
# 手动切换到从节点
pg_ctlcluster 17 main promote

# 更新连接字符串 / VIP / DNS
export POSTGRES_CONNECTION_STRING="postgresql://e2b:pass@postgres-replica:5432/e2b"
```

## 7. 监控建议

```text
推荐部署：
1. Prometheus + Grafana（指标）
2. Loki（日志聚合）
3. Alertmanager（告警）
```

关键监控项：

- API 请求延迟和错误率
- Orchestrator sandbox 数量和节点利用率
- Redis 内存使用、连接数、主从切换事件
- PostgreSQL 连接数、复制延迟、主从角色变化
- Nomad 集群健康状态
- 节点 CPU / 内存 / 磁盘 / hugepages / NBD 使用率
