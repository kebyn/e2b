# E2B 2026.28 用户可见功能参考

本文只记录可由固定上游标签 `2026.28`（commit `fda7bef1095afb909197e272c0a8a123797f0bfb`）验证、且会影响 SDK、HTTP API 或私有化入口的能力。完整契约仍以 `infra/spec/openapi.yml` 和对应 handler 为准。

## 1. Sandbox Host 与 Header 寻址

Client Proxy 支持两种目标表达方式：

| 入口 | 目标来源 | 适用场景 |
|------|----------|----------|
| `{port}-{sandboxID}.<DOMAIN_NAME>` | Host 最左侧 label | SDK 或调用方生成的普通 Host URL、浏览器和普通公网入口 |
| `sandbox.<DOMAIN_NAME>`、`localhost` 或 IP | `E2b-Sandbox-Id` + `E2b-Sandbox-Port` | 稳定共享域名、bridge、LAN、on-prem、直接 IP 入口 |

固定上游 `self-host.md` 验证了 SDK 的 `domain` 参数和 CLI 的 `E2B_DOMAIN` 用法；固定 `infra` 源码验证了 Client Proxy 接受两个 routing Header，但不能证明所有 SDK 版本都会自动注入它们。仅把 `E2B_SANDBOX_URL` 设为普通自定义 Host 时，如果 Host 没有编码 `{port}-{sandboxID}` 且请求未携带两个 Header，Client Proxy 无法解析目标；具体 SDK 行为必须查对应正式版本文档。

```bash
curl https://sandbox.e2b.example.com/ \
  -H 'E2b-Sandbox-Id: isv6ril5xadwn1k9t2jye' \
  -H 'E2b-Sandbox-Port: 8080'
```

共享 Host、localhost 或 IP 上只要出现任一 routing Header，两个 Header 就必须同时存在；缺失、无效端口或无效 Sandbox ID 返回 `400`。普通 `{port}-{sandboxID}` Host 始终是权威目标，请求 Header 不能覆盖它。

详细的代理兼容逻辑、DNS、TLS 和 Ingress 配置见 [不修改代码完整部署指南](../deployment/不修改代码完整部署指南.md#51-沙箱-url-构造原理)。

## 2. 流量触发暂停 Sandbox 恢复

创建 Sandbox 时可通过公开 API 保存自动恢复策略：

```bash
curl -X POST https://api.e2b.example.com/sandboxes \
  -H 'Content-Type: application/json' \
  -H 'X-API-Key: <api-key>' \
  -d '{
    "templateID": "base",
    "timeout": 300,
    "autoPause": true,
    "autoResume": {"enabled": true}
  }'
```

Client Proxy 在 Redis catalog 未命中时，只有同时满足以下条件才会尝试通过 API gRPC 恢复：

1. Sandbox 快照保存了 `autoResume.enabled=true`。
2. 快照不是 filesystem-only。
3. Client Proxy 配置了 `API_INTERNAL_GRPC_ADDRESS`，或配置了可用的 TLS/OIDC `API_EDGE_GRPC_ADDRESS` 回退通道。
4. 私有 ingress 或 secure envd 请求携带有效 access token；Client Proxy 会把对应 token 转交 API 校验。

`sandbox-auto-resume` Feature Flag 在 `2026.28` 已删除。当前行为由 Sandbox 的 `autoResume` API 配置和 Client Proxy 的 API gRPC 通道共同决定；两个 gRPC 地址都为空时，不会自动发现恢复通道。

Client Proxy 仅在 internal 地址为空时选择 Edge 地址，并对 Edge 连接强制 TLS；API 的 `API_EDGE_GRPC_PORT=5109` listener 自身是明文 gRPC，因此 Edge 地址必须指向外部 TLS 终止入口，由该入口转发到 `5109`。API 必须配置 `CLIENT_PROXY_OIDC_ISSUER_URL`，Client Proxy 必须同时配置三个 `API_EDGE_GRPC_OAUTH_*` 字段，token 还必须包含 `sandboxes:lifecycle` scope 和匹配目标集群组织的 `org_id`。缺少 API issuer 会使请求被拒绝；选择 Edge 通道时 OAuth 三字段部分配置会使 Client Proxy 启动失败，全部不配也不能得到可用的匿名 Edge 通道。

## 3. Sandbox 网络 transform 与 SOCKS5 出口

`POST /sandboxes` 的 `network` 字段和 `PUT /sandboxes/{sandboxID}/network` 支持：

| 字段 | 行为 |
|------|------|
| `allowOut` | CIDR、IP 或域名允许列表；允许规则优先于拒绝规则 |
| `denyOut` | CIDR 或 IP 拒绝列表；不接受域名 |
| `rules` | 按域名匹配的 HTTP/HTTPS Header 注入或覆盖规则 |
| `egressProxy` | `host:port` 格式的 SOCKS5 出口，可选 RFC 1929 用户名/密码 |

示例：

```json
{
  "templateID": "base",
  "network": {
    "allowOut": ["api.example.com"],
    "denyOut": ["0.0.0.0/0"],
    "rules": {
      "api.example.com": [
        {
          "transform": {
            "headers": {
              "X-Private-Route": "tenant-a"
            }
          }
        }
      ]
    },
    "egressProxy": {
      "address": "proxy.example.com:1080",
      "username": "sandbox-user",
      "password": "<runtime-secret>"
    }
  }
}
```

数据链路的责任边界如下：API 控制面负责 `egressProxy` 配置校验和 Feature Flag 门控；Orchestrator 网络层负责 allow/deny 以及 nftables/iptables 放行和 TCP 流量透明重定向；只有部署在 Orchestrator 节点上的 BYOP userspace `EgressProxy`/dialer 实现才负责 SOCKS5 协商、RFC 1929 认证、`CONNECT` 请求和隧道建立后的双向转发。这里的“Orchestrator 支持 SOCKS5”是节点级架构概括，不表示 Orchestrator API 或公共网络控制代码本身执行 `CONNECT`。

部署和调用时需要注意：

- `network-transform-rules` 和 `byop-proxy-enabled` 在生产 offline fallback 中都是 `false`，使用前必须通过 LaunchDarkly 定向启用；前者按 team context 求值，后者当前按全局 context 求值。
- 网络 transform 要求模板中的 envd 版本不低于 `0.5.13`；版本为空或过旧时 API 会要求重建模板。
- `rules` 中列出域名不会自动放行该域名；仍需用 `allowOut` 允许流量。上例使用 `denyOut: ["0.0.0.0/0"]` 后只放行明确允许的目标。
- transform Header 的值是 API 收到的普通字符串；若 SDK 支持 secret 引用，应在客户端解析后再发送。不要把真实密码或 token 写入仓库。
- allow/deny 在建立 SOCKS5 隧道之前执行。能够从 HTTP Host 或 TLS SNI 提取域名的流量，由 BYOP dialer 以 SOCKS5 `ATYP=DOMAIN` 发送目标域名，使代理端执行 DNS（remote DNS）；直接访问 IP，或 SSH 等无法提取域名的协议，则发送原始目标 IP。
- SOCKS5 代理地址必须是可解析的 `host:port`，端口范围为 `1..65535`，且所有解析出的地址都不能落入内部/受限 CIDR。用户名和密码各自最多 255 bytes；未设置用户名时也不能单独设置密码。
- 固定上游提交 `1fc3820d7` 主要完成 API、数据库、proto、配置校验和防火墙 plumbing，并为专用 BYOP 实现预留 `EgressProxy` 接口。固定 `2026.28` 公共源码中的 `NoopEgressProxy.SupportsBYOP()` 与 `tcpfirewall.Proxy.SupportsBYOP()` 都返回 `false`；Orchestrator 还会检查该能力。因此仅启用 `byop-proxy-enabled` 只会打开配置门控，不代表当前 Orchestrator 构建包含实际 SOCKS5 dialer，也不会使公共构建具备 `CONNECT` 和转发能力。
- `PUT /sandboxes/{sandboxID}/network` 是全量替换，不是 merge patch；省略某字段会清除其现有配置。

## 4. Filesystem-only 暂停和自动暂停

创建 Sandbox 时，`autoPauseMemory=false` 让超时自动暂停只保存文件系统，不保存内存：

```json
{
  "templateID": "base",
  "timeout": 300,
  "autoPause": true,
  "autoPauseMemory": false
}
```

也可在显式暂停时请求 filesystem-only 快照：

```bash
curl -X POST https://api.e2b.example.com/sandboxes/<sandbox-id>/pause \
  -H 'Content-Type: application/json' \
  -H 'X-API-Key: <api-key>' \
  -d '{"memory": false}'
```

恢复 filesystem-only 快照会从磁盘冷启动，内存状态、运行中进程和连接都会丢失。它必须显式 connect/resume，不能由任意入站流量自动恢复；因此 `autoPauseMemory=false` 必须与 `autoPause=true` 一起使用，也不能和 `autoResume.enabled=true` 组合。

## 5. 模板列表游标分页

`GET /v2/templates` 是新的模板列表接口；旧 `GET /templates` 在 OpenAPI 中已标记 deprecated。新接口接收 `limit`、`nextToken` 和可选 `teamID`，`limit` 默认及最大值都是 `100`。

```bash
curl -i 'https://api.e2b.example.com/v2/templates?limit=50' \
  -H 'X-API-Key: <api-key>'
```

响应 body 仍是模板数组。存在下一页时，服务端在响应 Header 返回 `X-Next-Token`；下一次请求把该值作为 `nextToken` 原样传回：

```text
GET /v2/templates?limit=50&nextToken=<X-Next-Token>
```

不要从 token 内容推断排序或自行构造 token。无下一页时不会返回 `X-Next-Token`。

## 6. Snapshot 名称过滤

`GET /snapshots` 新增可选 `name` query，可按 snapshot 名称或 ID 过滤，也接受带 namespace 或 tag 的形式：

```bash
curl --get https://api.e2b.example.com/snapshots \
  -H 'X-API-Key: <api-key>' \
  --data-urlencode 'name=my-team/my-snapshot:v1'
```

该接口仍支持 `sandboxID`、`limit` 和 `nextToken`；名称过滤不会改变游标分页契约。
