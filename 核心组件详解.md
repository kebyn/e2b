## 1. Supabase

### 是什么
Supabase 是一个开源的 Firebase 替代品，提供 PostgreSQL 数据库、认证、实时订阅等功能。在 E2B 中，Supabase **仅用于用户认证**。

### 在 E2B 中的用途

```
用户请求 → Supabase JWT Token → E2B API 验证 → 授权访问
```

1. **用户认证**：用户通过 Supabase Auth（邮箱/密码、OAuth、Magic Link 等）登录
2. **JWT 签发**：Supabase 为已认证用户签发 JWT Token
3. **E2B 验证**：E2B API 使用 `SUPABASE_JWT_SECRETS` 验证 Token 有效性
4. **用户识别**：从 JWT 中提取用户 ID，关联到 E2B 的 team/sandbox

### 工作流程

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Browser   │────▶│  Supabase   │────▶│   E2B API   │
│  (SDK)      │     │   (Auth)    │     │  (Verify)   │
└─────────────┘     └─────────────┘     └─────────────┘
      │                    │                    │
      │  1. Login          │                    │
      │───────────────────▶│                    │
      │                    │                    │
      │  2. JWT Token      │                    │
      │◀───────────────────│                    │
      │                    │                    │
      │  3. API Request + JWT                  │
      │─────────────────────────────────────────▶
      │                    │                    │
      │                    │  4. Verify JWT     │
      │                    │◀───────────────────│
      │                    │                    │
      │  5. Response       │                    │
      │◀─────────────────────────────────────────│
```

### 环境变量

| 变量 | 说明 |
|------|------|
| `SUPABASE_JWT_SECRETS` | JWT 签名密钥（支持多个，逗号分隔，用于密钥轮换） |

### 支持的认证方式

在 `packages/auth/pkg/auth/` 中定义：

```go
// 认证器类型
auth.NewApiKeyAuthenticator(...)          // X-API-Key header
auth.NewAccessTokenAuthenticator(...)     // Bearer token (非 Supabase)
auth.NewSupabaseTokenAuthenticator(...)   // Supabase JWT
auth.NewSupabaseTeamAuthenticator(...)    // Supabase + team ID
auth.NewAdminTokenAuthenticator(...)      // 管理员 token
```

### 为什么需要多个密钥

```go
// 支持密钥轮换：旧密钥签发的 token 仍然有效
SupabaseJWTSecrets []string `env:"SUPABASE_JWT_SECRETS"`
```

当轮换 JWT 密钥时：
1. 发布新密钥
2. 配置 `SUPABASE_JWT_SECRETS=old-secret,new-secret`
3. 等待旧 token 过期（通常 1 小时）
4. 移除旧密钥：`SUPABASE_JWT_SECRETS=new-secret`

## 2. Nomad

### 是什么
HashiCorp Nomad 是一个工作负载编排器，类似 Kubernetes。E2B 使用 Nomad 来**管理和调度 Orchestrator 节点**。

### 在 E2B 中的用途

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   E2B API   │────▶│    Nomad    │────▶│ Orchestrator│
│             │     │  (Cluster)  │     │   Nodes     │
└─────────────┘     └─────────────┘     └─────────────┘
      │                    │                    │
      │ 1. Query nodes     │                    │
      │───────────────────▶│                    │
      │                    │                    │
      │ 2. Node list       │                    │
      │◀───────────────────│                    │
      │                    │                    │
      │ 3. gRPC call to specific node          │
      │─────────────────────────────────────────▶
```

### 主要功能

1. **节点发现**：API 通过 Nomad 查询可用的 Orchestrator 节点
2. **工作分配**：Nomad 管理 Orchestrator 和 Template Manager 的部署
3. **健康检查**：Nomad 监控节点健康状态
4. **自动恢复**：节点故障时自动重启任务

### 使用 Nomad 的代码示例

```go
// packages/api/internal/orchestrator/client.go
func (o *Orchestrator) getNodes(ctx context.Context) ([]*node, error) {
    options := &nomadapi.QueryOptions{}
    // 从 Nomad 获取所有 Orchestrator 节点
    nomadNodes, _, err := o.nomadClient.Nodes().List(options.WithContext(ctx))
    // ...
}

// packages/api/internal/clusters/discovery/nomad.go
func ListOrchestratorAndTemplateBuilderAllocations(
    ctx context.Context,
    client *nomadapi.Client,
    filter NomadQueryFilter,
) ([]Allocation, error) {
    // 列出所有 Orchestrator 和 Template Builder 的分配
}
```

### 环境变量

| 变量 | 必需 | 默认值 | 说明 |
|------|------|--------|------|
| `NOMAD_ADDRESS` | ❌ | `http://localhost:4646` | Nomad API 地址 |
| `NOMAD_TOKEN` | ❌ | - | Nomad ACL Token（生产环境需要） |

### 本地开发模式

在本地开发时，可以跳过 Nomad：
- 不设置 `NOMAD_ADDRESS`，使用默认值
- API 会降级使用本地节点发现
- 适用于单节点测试

## 3. LaunchDarkly (Feature Flags)

### 是什么
LaunchDarkly 是一个功能开关（Feature Flags）服务，允许在**不重新部署代码**的情况下动态控制应用行为。

### 在 E2B 中的用途

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│  LaunchDarkly│     │  E2B API /  │     │  Runtime    │
│  (Dashboard) │────▶│  Orchestrator│────▶│  Behavior   │
└─────────────┘     └─────────────┘     └─────────────┘
      │                    │                    │
      │ 1. Set flag        │                    │
      │  max-sandboxes=50  │                    │
      │                    │                    │
      │                    │ 2. Read flag value │
      │                    │───────────────────▶│
      │                    │                    │
      │                    │ 3. Apply limit: max 50 sandboxes
```

## 4. Redis

### 是什么
Redis 是一个高性能的内存键值存储。E2B 使用 Redis 作为**分布式缓存和状态存储**。

### 在 E2B 中的用途

```
┌─────────────────────────────────────────────────────────────┐
│                         Redis                               │
├─────────────────────────────────────────────────────────────┤
│  sandbox:{id}:catalog  →  Sandbox 位置和状态（路由用）      │
│  sandbox:{id}:state    →  Sandbox 运行状态                  │
│  template:{id}:cache   →  模板缓存元数据                    │
│  snapshot:{id}:cache   →  快照缓存元数据                    │
│  team:{id}:sandboxes   →  团队的 sandbox 列表               │
│  events:sandbox        →  Sandbox 事件流（Streams）         │
│  rate:limit:{key}      →  速率限制计数器                    │
│  peer:registry         →  节点注册表（P2P 发现）            │
└─────────────────────────────────────────────────────────────┘
```

### 主要数据结构

#### 1. Sandbox Catalog（路由表）

```go
// 用于 Client Proxy 查找 sandbox 所在节点
type SandboxCatalogEntry struct {
    SandboxID       string    `json:"sandbox_id"`
    ExecutionID     string    `json:"execution_id"`
    Address         string    `json:"address"`      // Orchestrator 地址
    TemplateID      string    `json:"template_id"`
    TeamID          string    `json:"team_id"`
    // ...
}

// 存储: sandbox:catalog:{sandboxID}
// TTL: 与 sandbox 生命周期一致
```

#### 2. Sandbox 状态存储

```go
// 用于 API 管理 sandbox 状态
// Keys:
//   sandbox:state:{teamID}:{sandboxID}  →  sandbox 数据
//   sandbox:expiration                  →  过期时间 ZSet
//   team:sandboxes:{teamID}             →  team 的 sandbox 集合
```

#### 3. 模板/快照缓存

```go
// 用于 Orchestrator 缓存模板元数据
// 避免频繁查询数据库
cache.NewRedisCache[*TemplateInfo](RedisConfig[*TemplateInfo]{...})
```

#### 4. 事件流（Redis Streams）

```go
// 用于异步事件处理
event.NewRedisStreamsDelivery[SandboxEvent](redisClient, "sandbox-events")
```

#### 5. P2P 节点注册表

```go
// 用于 Orchestrator 之间发现彼此
peerclient.NewRedisRegistry(redisClient, nodeAddress)
```

### 架构图

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│ Client Proxy│────▶│    Redis    │◀────│ Orchestrator│
│             │     │  (Catalog)  │     │   Node A    │
└─────────────┘     └─────────────┘     └─────────────┘
      │                    ▲                    │
      │ Query sandbox      │                    │
      │ location           │ Register           │
      │                    │ sandbox            │
      ▼                    │                    ▼
┌─────────────┐            │            ┌─────────────┐
│   Sandbox   │            │            │   Sandbox   │
│   (HTTP)    │◀───────────┼───────────▶│   (gRPC)    │
└─────────────┘            │            └─────────────┘
                           │
                    ┌─────────────┐
                    │  Orchestrator│
                    │   Node B    │
                    └─────────────┘
```

## 5. Volume Token

### 是什么
Volume Token 是一个 JWT（JSON Web Token），用于**授权访问持久卷**。它类似于 AWS S3 的 Presigned URL，但用于 E2B 的持久卷系统。

### 在 E2B 中的用途

```
┌─────────────┐     ┌─────────────┐     ┌─────────────┐
│   Client    │────▶│   E2B API   │────▶│  Orchestrator│
│             │     │ (Issue JWT) │     │ (Verify JWT) │
└─────────────┘     └─────────────┘     └─────────────┘
      │                    │                    │
      │ 1. Create volume   │                    │
      │───────────────────▶│                    │
      │                    │                    │
      │ 2. Volume Token (JWT)                  │
      │◀───────────────────│                    │
      │                    │                    │
      │ 3. Mount volume + Token                │
      │─────────────────────────────────────────▶
      │                    │                    │
      │                    │  4. Verify token   │
      │                    │◀───────────────────│
      │                    │                    │
      │ 5. Access granted  │                    │
      │◀─────────────────────────────────────────│
```

### Token 结构

```json
{
  // JWT Header
  "alg": "ES256",
  "typ": "JWT",
  "tokid": "local-dev-2026-03-03"  // 密钥名称

  // JWT Claims
  {
    // 标准字段
    "aud": "cluster-id",           // 受众（集群 ID）
    "exp": 1711334400,             // 过期时间
    "iat": 1711330800,             // 签发时间
    "iss": "local.e2b.dev",        // 签发者
    "jti": "unique-token-id",      // 唯一 ID
    "nbf": 1711330800,             // 生效时间
    "sub": "team-uuid",            // 主题（团队 ID）

    // 自定义字段
    "teamid":  "team-uuid",        // 团队 ID
    "volid":   "volume-uuid",      // 卷 ID
    "voltype": "persistent"        // 卷类型
  }
}
```

### 签发流程

```go
// packages/api/internal/handlers/volume_token.go
func generateVolumeContentToken(config cfg.VolumesTokenConfig, volume queries.Volume, team *types.Team) (string, error) {
    claims := jwt.MapClaims{
        "aud":     clusterID.String(),
        "exp":     jwt.NewNumericDate(expiration),
        "iat":     jwt.NewNumericDate(now),
        "iss":     config.Issuer,
        "jti":     uuid.NewString(),
        "nbf":     jwt.NewNumericDate(now),
        "sub":     team.ID.String(),
        "teamid":  team.ID.String(),
        "volid":   volume.ID.String(),
        "voltype": volume.VolumeType,
    }

    token := jwt.NewWithClaims(config.SigningMethod, claims)
    token.Header["tokid"] = config.SigningKeyName
    return token.SignedString(config.SigningKey)
}
```

### 密钥类型说明

| 类型 | 算法 | 密钥格式 | 适用场景 |
|------|------|----------|----------|
| `ECDSA` | ES256 (P-256) | PEM 私钥 | **推荐**：体积小、性能好 |
| `RS256` | RSA 2048+ | PEM 私钥 | 兼容性好，体积大 |
| `HMAC` | HS256 | 任意字节 | 简单，需安全分发 |
| `EdDSA` | Ed25519 | PEM 私钥 | 最新最快，部分系统不支持 |

### 密钥生成示例

```bash
# ECDSA（推荐）
openssl ecparam -name prime256v1 -genkey -noout -out private.pem
openssl ec -in private.pem -pubout -out public.pem

# RSA
openssl genrsa -out private.pem 2048
openssl rsa -in private.pem -pubout -out public.pem

# 配置环境变量
VOLUME_TOKEN_SIGNING_KEY="ECDSA:$(base64 -w0 private.pem)"
```

### 环境变量

| 变量 | 必需 | 默认值 | 说明 |
|------|------|--------|------|
| `VOLUME_TOKEN_ISSUER` | ✅ | - | 签发者标识，如 `e2b.dev` |
| `VOLUME_TOKEN_SIGNING_METHOD` | ✅ | - | JWT 签名算法 |
| `VOLUME_TOKEN_SIGNING_KEY` | ✅ | - | 签名密钥，格式：`TYPE:base64(PEM)` |
| `VOLUME_TOKEN_SIGNING_KEY_NAME` | ✅ | - | 密钥名称，用于密钥轮换 |
| `VOLUME_TOKEN_DURATION` | ❌ | `1h` | Token 有效期 |

### 密钥轮换流程

```bash
# 1. 生成新密钥
openssl ecparam -name prime256v1 -genkey -noout -out new-key.pem

# 2. 更新配置（同时保留旧密钥用于验证）
VOLUME_TOKEN_SIGNING_KEY="ECDSA:$(base64 -w0 new-key.pem)"
VOLUME_TOKEN_SIGNING_KEY_NAME="prod-2026-03-25"

# 3. 等待旧 token 过期（默认 1 小时）

# 4. 验证新 token 正常工作后，移除旧配置
```

### 安全考虑

1. **密钥存储**：生产环境应使用 GCP Secrets Manager 或类似服务
2. **有效期控制**：`VOLUME_TOKEN_DURATION` 应根据使用场景调整
3. **集群隔离**：`aud` 字段验证 token 只在特定集群有效
4. **团队隔离**：`teamid` 字段确保 token 只访问团队自己的卷

### 组件职责总结

| 组件 | 职责 | 是否必须 |
|------|------|----------|
| **Supabase** | 用户认证、JWT 签发 | ❌ 可用 API Key 替代 |
| **Nomad** | 节点发现、任务编排 | ❌ 本地开发可跳过 |
| **LaunchDarkly** | 功能开关、动态配置 | ❌ 未配置时使用默认值 |
| **Redis** | 缓存、状态存储、事件流 | ❌ 未配置时降级为内存 |
| **Volume Token** | 持久卷访问授权 | ✅ 使用持久卷时必须 |
