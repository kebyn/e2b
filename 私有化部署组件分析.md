# 多节点高可用部署组件必要性分析

---

## 总览

| 组件 | 单节点 | 多节点(HA) | 私有化部署 | 需要修改代码? |
|------|--------|------------|------------|---------------|
| **Supabase** | ❌ 可不用 | ⚠️ 需要替代 | ✅ 必须替换 | ⚠️ 替换认证层 |
| **Nomad** | ❌ 可不用 | ✅ 必须 | ⚠️ 可选替换 | ✅ 需要适配 |
| **Consul** | ❌ 可不用 | ⚠️ 需要替代 | ⚠️ 可选替换 | ⚠️ 替换存储层 |
| **LaunchDarkly** | ❌ 可不用 | ❌ 可不用 | ❌ 可不用 | ❌ 无需修改 |
| **Redis** | ❌ 可不用 | ✅ 必须 | ✅ 必须 | ❌ 无需修改 |
| **ClickHouse** | ❌ 可不用 | ⚠️ 推荐 | ⚠️ 可选 | ❌ 无需修改 |

## 1. Supabase

### 当前用途
- **仅用于用户认证**（JWT 签发和验证）
- 不使用 Supabase 的数据库、存储等其他功能

### 代码依赖分析

```go
// packages/api/internal/handlers/store.go
authService := sharedauth.NewAuthService[*types.Team](
    authStore,
    authCache,
    config.SupabaseJWTSecrets,  // 仅用于验证 JWT
)

// packages/auth/pkg/auth/jwt.go
func ParseUserIDFromToken(ctx context.Context, secrets []string, supabaseToken string) (uuid.UUID, error) {
    claims, err := GetJWTClaims(ctx, secrets, supabaseToken)  // 标准 JWT 验证
    // ...
}
```

### 多节点 HA 必要性

| 场景 | 是否需要 | 原因 |
|------|----------|------|
| 单节点开发 | ❌ | 可用 API Key 代替 |
| 多节点生产 | ⚠️ | 需要统一用户认证 |
| 私有化部署 | ⚠️ | 可替换为其他 JWT 提供者 |

### 私有化替换方案

#### 方案 A：替换为自建 JWT 认证（推荐）

```
修改范围：packages/auth/pkg/auth/
```

```go
// 新增: packages/auth/pkg/auth/jwt_custom.go
// 使用标准 JWT 验证，不依赖 Supabase SDK

type JWTConfig struct {
    Issuer     string   // 自建认证服务的 issuer
    JWKSUrl    string   // 公钥地址，如 https://auth.example.com/.well-known/jwks.json
    Secrets    []string  // 或直接配置密钥
}

func NewCustomJWTAuth(config JWTConfig) AuthService {
    // 标准 JWT 验证，不依赖 Supabase
}
```

#### 方案 B：完全移除用户认证

```go
// 仅使用 API Key 认证，不支持用户登录
auth.NewApiKeyAuthenticator(...)      // 保留
auth.NewAccessTokenAuthenticator(...) // 保留
auth.NewAdminTokenAuthenticator(...)  // 保留
// 移除 Supabase 相关认证器
```

**环境变量变更：**
```bash
# 原有
SUPABASE_JWT_SECRETS=xxx

# 替换为
AUTH_JWT_ISSUER=https://your-auth-service.com
AUTH_JWKS_URL=https://your-auth-service.com/.well-known/jwks.json
# 或
AUTH_JWT_SECRETS=your-secret-key
```

### 当前用途
1. **节点发现**：API 查询可用的 Orchestrator 节点
2. **任务调度**：管理 Orchestrator/Template Manager 部署
3. **健康检查**：监控节点状态

### 代码依赖分析

```go
// packages/api/internal/orchestrator/client.go
func (o *Orchestrator) getNodes(ctx context.Context) ([]*node, error) {
    nomadNodes, _, err := o.nomadClient.Nodes().List(options.WithContext(ctx))
    // 从 Nomad 获取节点列表

// packages/api/internal/clusters/discovery/local.go
func (sd *LocalServiceDiscovery) Query(ctx context.Context) ([]Item, error) {
```

### 多节点 HA 必要性

| 场景 | 是否需要 | 原因 |
|------|----------|------|
| 单节点开发 | ❌ | 使用本地静态发现 |
| 多节点生产 | ✅ | 需要动态节点发现和调度 |
| 私有化部署 | ⚠️ | 可替换为 K8s 或静态配置 |

### 私有化替换方案

## 3. Consul

### 当前用途
- **仅用于网络 Slot 存储**（IP 地址分配的分布式锁）

### 代码依赖分析

```go
// packages/orchestrator/internal/sandbox/network/storage_kv.go
type StorageKV struct {
    consulClient *consulApi.Client  // 用于 IP Slot 分配
}

func NewStorageKV(nodeID string, config Config) (*StorageKV, error) {
    consulToken := utils.RequiredEnv("CONSUL_TOKEN", "...")
    consulClient, err := newConsulClient(consulToken)
    // ...
}
```

### 降级机制（已有）

```go
// packages/orchestrator/main.go
func newStorage(ctx context.Context, nodeID string, config network.Config) (network.Storage, error) {
    // 开发环境或配置为本地时，使用本地存储
    if env.IsDevelopment() || config.UseLocalNamespaceStorage {
        return network.NewStorageLocal(ctx, config)  // 文件系统 + 内存
    }
    // 生产环境使用 Consul KV
    return network.NewStorageKV(nodeID, config)
}
```

### 多节点 HA 必要性

| 场景 | 是否需要 | 原因 |
|------|----------|------|
| 单节点开发 | ❌ | 使用本地存储 |
| 多节点生产 | ⚠️ | 仅当多节点共享 IP 池时需要 |
| 私有化部署 | ⚠️ | 可替换为 Redis |

### 私有化替换方案

#### 方案 A：替换为 Redis（推荐）

```go
// 新增: packages/orchestrator/internal/sandbox/network/storage_redis.go
type StorageRedis struct {
    client redis.UniversalClient
    nodeID string
}

func (s *StorageRedis) Acquire(ctx context.Context) (*Slot, error) {
    // 使用 Redis SETNX 实现分布式锁
    for i := 0; i < 10; i++ {
        slotIdx := rand.Intn(s.slotsSize)
        key := fmt.Sprintf("ip-slot:%s:%d", s.nodeID, slotIdx)
        
        ok, err := s.client.SetNX(ctx, key, "1", 0).Result()
        if err == nil && ok {
            return NewSlot(key, slotIdx, s.config)
        }
    }
    return nil, fmt.Errorf("no slots available")
}
```

#### 方案 B：每节点独立 IP 池（无需共享存储）

```bash
# 节点 1: IP 池 10.0.1.0/24
# 节点 2: IP 池 10.0.2.0/24
# 节点 3: IP 池 10.0.3.0/24

# 环境变量
USE_LOCAL_NAMESPACE_STORAGE=true  # 使用本地存储
```

**环境变量变更：**
```bash
# 原有
CONSUL_TOKEN=xxx

# 替换为
IP_SLOT_STORAGE=redis  # 或 local
REDIS_URL=redis://localhost:6379
```

## 4. LaunchDarkly

### 当前用途
- **功能开关**：控制 40+ 个运行时参数
- **动态配置**：无需重启调整性能参数
- **灰度发布**：按 team/template 控制功能

### 代码依赖分析

```go
// packages/shared/pkg/featureflags/client.go
func NewClient() (*Client, error) {
    if launchDarklyApiKey == "" {
        // 离线模式：使用代码中的默认值
        return NewClientWithDatasource(launchDarklyOfflineStore)
    }
    // 在线模式：连接 LaunchDarkly
    ldClient, err := ldclient.MakeClient(launchDarklyApiKey, waitForInit)
    return &Client{ld: ldClient}, nil
}
```

### 降级机制（已内置）

```go
// 所有 flag 都有 fallback 默认值
var MaxSandboxesPerNode = newIntFlag("max-sandboxes-per-node", 200)  // 默认值

// 未配置 LAUNCH_DARKLY_API_KEY 时自动使用默认值
```

### 多节点 HA 必要性

| 场景 | 是否需要 | 原因 |
|------|----------|------|
| 单节点开发 | ❌ | 使用默认值 |
| 多节点生产 | ❌ | 使用默认值即可 |
| 私有化部署 | ❌ | 无需修改，使用默认值 |

### 私有化方案

**无需任何修改！** 代码已内置完整降级机制。

如需动态配置，可选方案：

#### 方案 A：配置文件（简单）

```yaml
# /etc/e2b/feature-flags.yaml
max-sandboxes-per-node: 100
sandbox-metrics-write: true
host-stats-enabled: true
```

#### 方案 B：环境变量覆盖

```bash
# 每个 flag 对应一个环境变量（需要少量代码修改）
E2B_FLAG_MAX_SANDBOXES_PER_NODE=100
E2B_FLAG_SANDBOX_METRICS_WRITE=true
```

#### 方案 C：自建 Feature Flag 服务

可使用开源替代：
- **Unleash**：轻量级，API 兼容
- **Flagsmith**：功能完整，有 UI
- **Flipt**：高性能，gRPC 支持

## 5. Redis

### 当前用途
1. **Sandbox Catalog**：路由表（sandbox → orchestrator 映射）
2. **状态存储**：sandbox 生命周期状态
3. **模板缓存**：减少数据库查询
4. **事件流**：异步事件处理
5. **P2P 注册表**：节点发现
6. **速率限制**：API 限流

### 降级机制（已内置）

```go
// packages/client-proxy/main.go
if redisClient != nil {
    catalog = NewRedisSandboxesCatalog(redisClient, featureFlagsClient)
} else {
    catalog = NewMemorySandboxesCatalog()  // 内存降级
}

// packages/api/internal/orchestrator/orchestrator.go
if redisClient != nil {
    routingCatalog = e2bcatalog.NewRedisSandboxesCatalog(redisClient, featureFlags)
} else {
    routingCatalog = e2bcatalog.NewMemorySandboxesCatalog()
}
```

### 多节点 HA 必要性

| 场景 | 是否需要 | 原因 |
|------|----------|------|
| 单节点开发 | ❌ | 内存降级 |
| 多节点生产 | ✅ | 跨节点共享状态 |
| 私有化部署 | ✅ | 必须部署 Redis |

### 私有化部署建议

**无需修改代码**，只需部署 Redis：

```bash
# 单节点 Redis（简单部署）
docker run -d --name redis redis:7-alpine

# 或 Redis Sentinel（高可用）
docker run -d --name redis-sentinel redis:7-alpine redis-sentinel

# 环境变量
REDIS_URL=redis://redis-host:6379
# 或集群模式
REDIS_CLUSTER_URL=redis-cluster-host:6379
```

## 代码修改优先级

### 必须修改（多节点部署）

| 组件 | 修改内容 | 工作量 |
|------|----------|--------|
| **Nomad** | 替换节点发现机制 | 中等 |
| **Redis** | 无需修改，直接部署 | 无 |

### 建议修改（私有化安全）

| 组件 | 修改内容 | 工作量 |
|------|----------|--------|
| **Supabase** | 替换为自建 JWT 认证 | 中等 |
| **Consul** | 替换为 Redis 或本地存储 | 小 |

### 无需修改

| 组件 | 原因 |
|------|------|
| **LaunchDarkly** | 已内置离线模式 |
| **ClickHouse** | 已内置 Noop 降级 |
