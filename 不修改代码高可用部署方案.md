## 1. 核心结论

| 组件 | 是否必须 | 降级方案 | 影响 |
|------|----------|----------|------|
| **Nomad** | ✅ **必须** | 无降级 | 多节点 Orchestrator 发现依赖此组件 |
| **Redis** | ✅ **必须** | 内存降级仅适合单实例 | 无 Redis 无法跨 API 实例共享 sandbox 状态 |
| **PostgreSQL** | ✅ **必须** | 无降级 | 核心数据存储 |
| **Consul** | ❌ 可选 | 本地存储 | 每节点独立 IP 池时可省略 |
| **LaunchDarkly** | ❌ 可选 | 默认值 | 功能开关不可动态调整 |
| **ClickHouse** | ❌ 可选 | Noop | 无监控指标 |
| **Supabase** | ❌ 可选 | API Key | 无用户级认证 |

## 2. 高可用拓扑总览

```text
用户流量
  -> LB / Nginx / HAProxy
  -> API (3 实例) + Client Proxy (2+ 实例)
  -> Nomad 发现多个 Orchestrator 节点
  -> Redis Sentinel / PostgreSQL 主从提供共享状态与持久化
```

### 推荐最小实例数

| 组件 | 最低建议 | 说明 |
|------|----------|------|
| Nomad Server | 3 | 保证仲裁 |
| Redis Sentinel | 3 | 自动主从切换 |
| PostgreSQL | 1 主 + 1 从 | 至少具备切换能力 |
| API | 3 | LB 后无状态扩缩容 |
| Client Proxy | 2+ | 保障 sandbox 访问入口 |
| Orchestrator | N 台计算节点 | 由 Nomad system job 管理 |

## 4. 高可用关注点

### 4.1 API 层

- API 必须多实例部署在负载均衡后
- `SANDBOX_STORAGE_BACKEND` 多实例场景必须为 `redis`
- 健康检查必须接入 LB 剔除机制
- 所有实例的 `VOLUME_TOKEN_*`、数据库、Redis 配置必须一致

### 4.2 Client Proxy 层

- 至少 2 个实例
- 保证 wildcard sandbox 域名入口不单点
- Proxy 的 Redis 连接必须指向共享状态存储

### 4.3 Orchestrator 层

- 通过 Nomad system job 运行在多个计算节点
- 节点故障后要能被重新调度或替换
- 建议每节点独立 IP 池，避免引入额外 Consul 复杂度
- 节点资源（NBD、hugepages、磁盘）要按峰值容量预留

### 4.4 数据层

- Redis 需要 Sentinel 或等价自动主从切换能力
- PostgreSQL 至少准备一个可提升的副本
- 必须做主从切换演练，而不是只看“配置存在”

## 5. 高可用验证

### 5.1 健康检查

```bash
# API 健康检查
curl http://api-1:3000/health
curl http://api-2:3000/health
curl http://api-3:3000/health

# Client Proxy 健康检查
curl http://proxy-1:3003/
curl http://proxy-2:3003/

# Orchestrator 健康检查（或通过 Nomad）
nomad status e2b-orchestrator
nomad node status
```

### 5.2 Nomad 状态检查

```bash
# 查看 Nomad 集群状态
nomad server members

# 查看 Orchestrator 任务状态
nomad status e2b-orchestrator

# 查看节点分配
nomad node status -verbose
```

### 5.3 Redis 检查

```bash
# Redis 主从状态
redis-cli -h redis-master INFO replication

# Sentinel 状态
redis-cli -h sentinel-1 -p 26379 SENTINEL master mymaster
redis-cli -h sentinel-1 -p 26379 SENTINEL replicas mymaster
```

### 5.4 PostgreSQL 检查

```bash
# 主库可用性
psql -h postgres -U e2b -c "SELECT 1;"

# 查看复制状态（主库执行）
psql -h postgres -U postgres -c "SELECT * FROM pg_stat_replication;"
```

### 5.5 业务验证

```bash
# 1. 创建 sandbox
# 2. 访问 sandbox URL
# 3. 重启一个 API 实例后再次访问
# 4. 验证会话与路由未丢失
```

## 6. 故障恢复

### 6.1 API 节点故障

```bash
# 自动: 负载均衡器检测健康检查失败后自动剔除
# 手动: 重启服务
systemctl restart e2b-api
```

### 6.2 Orchestrator 节点故障

```bash
# Nomad 自动重新调度到其他节点
# 或手动在新节点启动 / 恢复节点
nomad job restart e2b-orchestrator
```

### 6.3 Redis 故障

```bash
# Sentinel 自动故障转移
# 检查新主节点
redis-cli -h sentinel-1 -p 26379 SENTINEL get-master-addr-by-name mymaster
```

### 6.4 PostgreSQL 故障

```bash
# 手动切换到从节点
pg_ctlcluster 17 main promote

# 更新连接字符串 / VIP / DNS
export POSTGRES_CONNECTION_STRING="postgresql://e2b:pass@postgres-replica:5432/e2b"
```

## 7. 监控建议

```text
推荐部署：
1. Prometheus + Grafana（指标）
2. Loki（日志聚合）
3. Alertmanager（告警）
```

关键监控项：

- API 请求延迟和错误率
- Orchestrator sandbox 数量和节点利用率
- Redis 内存使用、连接数、主从切换事件
- PostgreSQL 连接数、复制延迟、主从角色变化
- Nomad 集群健康状态
- 节点 CPU / 内存 / 磁盘 / hugepages / NBD 使用率
